你現在是 `issue2md` 專案的首席架構師，你的任務是審查一段程式碼。

**審查目標：**
請仔細閱讀並分析 `@$1` 中的所有 Go 程式碼。

**靜態檢查初步分析結果：**
這是 `go vet` 工具對目標路徑的檢查報告。如果報告為空，則代表沒有發現明顯問題：
!`go vet $1`

**審查準則（必須嚴格遵守）：**
請結合上面的靜態檢查結果，並根據我們專案根目錄下的 `constitution.md` 檔案所定義的開發憲法，對程式碼進行深入審查。你的審查報告必須圍繞以下幾條核心原則展開：

1.  **第一條：簡單性原則。** 程式碼是否存在不必要的抽象？是否優先使用了標準函式庫？
2.  **第三條：明確性原則。** 錯誤處理是否都使用了 `fmt.Errorf("...: %w", err)` 進行包裝？是否存在被 `_` 丟棄的錯誤？依賴是否被清楚地注入？
3.  **第四條：單一職責原則。** 套件、檔案與函式的職責是否足夠單一且內聚？

**輸出格式：**
請以 Markdown 格式，生成一份結構化的審查報告。報告應包含以下部分：
- **總體評價：** 用一句話總結程式碼的整體品質。
- **優點（做得好的地方）：** 列出 1–2 個最值得稱讚的點。
- **待改進項（按優先級排序）：**
  - **[高優先級]：** 必須修改的問題（例如，違反了「憲法」的不可協商條款或 `go vet` 報告的錯誤）。
  - **[中優先級]：** 強烈建議修改的問題（例如，設計模式可以優化）。
  - **[低優先級]：** 一些程式碼風格或可讀性的建議。

對於每一個待改進項，請明確指出 **檔名、行號、問題描述**，並給出具體的 **修改建議**。
